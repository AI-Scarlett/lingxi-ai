# 📋 灵犀版本号规范

> **语义化版本 + 快速迭代** 🚀

---

## 版本号格式

```
v主版本。次版本.修订版本
  │    │    │
  │    │    └─ 优化/修复 (Patch)
  │    └────── 新功能 (Minor)
  └─────────── 重大变更 (Major)
```

---

## 版本迭代规则

### 🆕 第二位版本号（次版本）- 新功能

**触发条件：**
- 新增核心功能模块
- 新增平台集成
- 新增系统（语音、多平台等）

**示例：**
- v2.4.0 → v2.5.0：新增多平台集成（飞书 + 钉钉 + 企业微信）
- v2.3.0 → v2.4.0：新增语音交互系统
- v2.2.0 → v2.3.0：新增智能学习系统

---

### 🔧 第三位版本号（修订版本）- 优化/修复

**触发条件：**
- 性能优化
- Bug 修复
- 代码重构
- 文档更新
- 响应流程优化

**示例：**
- v2.5.0 → v2.5.1：性能优化 + 缺陷修复 + 响应流程优化
- v2.3.0 → v2.3.1：Bug 修复
- v2.2.0 → v2.2.1：性能优化

---

### ⚠️ 第一位版本号（主版本）- 重大变更

**触发条件：**
- 架构重构
- 破坏性变更
- 不向后兼容

**示例：**
- v1.0.0 → v2.0.0：异步任务系统重构

---

## 发布流程

### 每次更新必须完成：

1. **更新 CHANGELOG.md**
   - 添加新版本更新日志
   - 说明核心功能/优化/修复
   - 标注性能提升数据

2. **更新 README.md**
   - 更新版本号徽章
   - 添加新版本核心特性到顶部
   - 保持版本历史完整

3. **Git 提交**
   ```bash
   git add -A
   git commit -m "feat(v2.X.0): 新功能描述" 
   # 或
   git commit -m "fix(v2.5.X): 修复/优化描述"
   ```

4. **打 Tag**
   ```bash
   git tag -a v2.X.X -m "灵犀 v2.X.X - 版本说明"
   ```

5. **推送 GitHub**
   ```bash
   git push origin main
   git push origin v2.X.X
   ```

6. **创建 Release**
   - GitHub Releases 页面
   - 复制 CHANGELOG 内容
   - 添加下载链接

```
- path/to/new/file.py

更新文件:
- README.md
- CHANGELOG.md
```

### 优化/修复（修订版本）
```
fix(v2.5.X): 修复/优化描述

- 优化项 1
- 修复项 2
- 性能提升数据

更新文件:
- path/to/file.py
- CHANGELOG.md
```

## 快速迭代原则

**7 天 9 个版本是我们的节奏！** 🚀

- 新功能 → 当天开发、当天测试、当天发布
- 优化修复 → 发现即修复，不过夜
- 文档同步 → 代码更新的同时文档必须更新
- Tag 必打 → 每个版本都要有 Git Tag
