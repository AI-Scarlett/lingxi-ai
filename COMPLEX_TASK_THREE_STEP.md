## 📚 核心思想

**用最少的资源识别真正需要资源的任务，然后把资源集中在那些任务上。**

---

## 🏗️ S0→S1→S2→S3 四层架构

### S0: 零成本预筛选

**特点：**
- 🎯 规则匹配，0 token
- ⚡ 过滤 ~80% 消息
- ✅ 白名单直接放行

**白名单（直接执行）：**
- 单轮问答："几点了"、"天气怎样"
- 延续指令："继续"、"下一步"
- 简单指令："搜索 X"、"发消息给 Y"
- 闲聊确认："好的"、"谢谢"

**触发信号（进入 S1）：**
| 信号 | 示例 |
|------|------|
| 长度 | > 200 字 |
| 意图动词 | 开发、构建、设计、部署、迁移、重构 |
| 范围词 | 整个、全部、系统、架构、从零开始 |
| 多步模式 | "先…然后…最后…" |
| 不确定性 | 代理无法判断单步执行路径 |
| 显式触发 | "复杂任务"、"三步法"、"需要规划" |

---

### S1: 轻量复杂度评估

**五维打分（每项 1-5 分）：**

| 维度 | 评估内容 |
|------|---------|
| 步骤数 | 预估需要几步 |
| 知识域 | 涉及几个不同领域 |
| 不确定性 | 有多少未知需要先搜索 |
| 失败代价 | 做错了的回退成本 |
| 工具链 | 需要协调几个工具/系统 |

**决策阈值：**
- **≤ 8 分** → 直接执行
- **9-15 分** → 轻规划（列步骤，边做边调整）
- **> 15 分** → 完整三步法（S2 → S3）

**设计哲学：**
> 宁可多评估一些，也不要漏掉真正复杂的任务

---

### S2: 深度规划 & 审计

**Plan Mode（高能力模型）：**
- 任务分解为 DAG 结构
- 标注依赖和风险
- 输出执行蓝图

**Audit Mode（审计模型）：**
- 检查完整性
- 检查合理性
- 检查可行性
- 最多 2 轮 Plan-Audit 修改循环

**DAG 并行步骤示例：**
```json
{
  "steps": [
    {"id": 1, "action": "分析需求", "depends_on": []},
    {"id": "2a", "action": "搜索文档", "depends_on": [1]},
    {"id": "2b", "action": "查本地缓存", "depends_on": [1]},
    {"id": 3, "action": "综合结果", "depends_on": ["2a", "2b"]}
  ]
}
```

**执行蓝图锁定：**
> 规划是铁约束，不是参考建议

---

### S3: 分阶段执行 & 质量控制

**执行流程：**
1. 按 Phase 执行，同 Phase 内步骤可并行
2. 每步完成后 QA 审计 → 通过则成果锁定
3. 后续 Phase 使用前置 Phase 的锁定成果

**三道防线：**
- **Audit** - 每步完成后审计
- **QA** - 质量检查
- **Defect Rule** - 缺陷修改分级

**缺陷修改分级：**
| 严重度 | 处理方式 |
|--------|---------|
| Critical | 自动批准 |
| High | 自动批准 + 通知人类 |
| Medium | 人类确认后修改 |
| Low | QA 自行决定 |

## 🎯 设计原则

### 1. 分层防御，而非一刀切

灵感来自网络安全领域的**纵深防御（Defense in Depth）**：
- S0 是门卫 - 规则匹配，零成本
- S1 是初诊医生 - 快速检查，给出评分
- S2 是手术规划 - 深度分析，锁定蓝图
- S3 是手术执行 - 分阶段，有质量控制

**每一层只处理上一层无法判定的案例。**

---

### 2. 灵敏度 - 成本的帕累托平衡

**两种错误：**
- **漏报（False Negative）**：复杂任务被当作简单任务执行 → 失败、返工、浪费
- **误报（False Positive）**：简单任务被当作复杂任务评估 → 多花几百 token，但无害

**设计哲学：**
> 漏报的代价远大于误报。宁可多评估一些，也不要漏掉真正复杂的任务。

---

### 3. 规划是铁约束，不是参考建议

借鉴工程领域的**"设计冻结"（Design Freeze）**概念：
- S2 产出的执行蓝图被设计为**锁定的**
- S3 的所有执行都围绕它展开
- 偏离必须记录原因

> 无纪律的变更比不完美的计划更危险。

---

### 4. 并行是天然的，串行是例外

- S2 的步骤规划使用**DAG（有向无环图）**结构
- 每个步骤声明自己的依赖（depends_on）
- 执行引擎自动并行调度所有依赖已满足的步骤

> 这不是优化——这是还原任务的真实结构。

---

### 5. 质量不是事后检查，而是贯穿全程

- 每个步骤完成时就审计
- 问题越早发现，修复成本越低
- 缺陷修改分级是权限控制

## 📊 完整流程图

```
用户消息
  ↓
┌──────────────────────────────────────────┐
│ S0 零成本预筛选                           │
│ 规则匹配 · 0 token · 过滤 ~80% 消息        │
│ ✓ 简单 → 直接执行                         │
│ ? 疑似复杂 → 进入 S1                       │
└──────────────┬───────────────────────────┘
               │ ~20% 消息
               ↓
┌──────────────────────────────────────────┐
│ S1 轻量复杂度评估                          │
│ 五维打分 · ~300 token                     │
│ ≤ 8 分 → 直接执行                          │
│ 9-15 分 → 轻规划，边做边调整                │
│ > 15 分 → 进入 S2                          │
└──────────────┬───────────────────────────┘
               │ ~5% 消息
               ↓
┌──────────────────────────────────────────┐
│ S2 深度规划 & 审计                         │
│ Plan Mode → Audit Mode → ≤2 轮修改         │
│ DAG 并行步骤 · 执行蓝图锁定                 │
└──────────────┬───────────────────────────┘
               │
               ↓
┌──────────────────────────────────────────┐
│ S3 分阶段执行 & 质量控制                   │
│ Phase 并行 (DAG) · QA 审计循环             │
│ 成果锁定 · 缺陷修改分级                    │
│ 三道防线：Audit → QA → Defect Rule        │
└──────────────┬───────────────────────────┘
               │
               ↓
             ✅ 完成

动态升级兜底：任何阶段执行中发现复杂度被低估，可中途触发重新评估并升级到完整流程。
```

## 🎓 设计哲学小结

> **用最少的资源识别真正需要资源的任务，然后把资源集中在那些任务上。**

这不是一个复杂的系统。它是一个对"什么时候该复杂、什么时候该简单"这个问题的系统性回答。

- 大多数消息不需要规划。少数消息需要精密规划。区分它们的成本应该接近于零。**这就是 S0 存在的意义。**
- 规划不是浪费时间。没有规划的执行才是。**这就是 S2 存在的意义。**
- 执行不是一个人的事。它需要审计、质量控制、和分级的问题修复机制。**这就是 S3 存在的意义。**
